## 快速开始

### ⚠️ 不要直接 Clone 本仓库！

你对本仓库**没有直接写入权限**，也**不应该**直接 clone 后往 `main` 分支推送。正确的做法是：

```
┌──────────────────────────────────────────────────────┐
│  ① Fork → ② Clone 你自己的 Fork → ③ 创建个人目录    │
│  ④ 写代码/文档 → ⑤ 提交 PR 到本仓库 main 分支       │
└──────────────────────────────────────────────────────┘
```

#### 第一步：Fork 本仓库

1. 点击本仓库右上角的 **Fork** 按钮
2. 在弹出页面中选择 Fork 到你自己的 GitHub 账号下
3. 等待几秒，你会在自己的账号下得到一个副本仓库（如 `你的用户名/saike-develop`）

#### 第二步：Clone 你的 Fork

```bash
# 将 <你的用户名> 替换为你自己的 GitHub 用户名
git clone https://github.com/<你的用户名>/saike-develop.git
cd saike-develop
```

#### 第三步：设置上游同步

```bash
# 添加本仓库为 upstream（上游），方便后续同步
git remote add upstream https://github.com/BobcGn/saike-develop.git

# 验证 remote 配置
git remote -v
# 你应该看到：
# origin    https://github.com/<你的用户名>/saike-develop.git (fetch/push)
# upstream  https://github.com/BobcGn/saike-develop.git (fetch/push)
```

## 目录结构规范

### 仓库整体结构

```
saike-develop/
├── .github/                    # CI 流水线配置（禁止修改）
├── .gitignore                  # 全局忽略规则（禁止修改）
├── README.md                   # 本文件（禁止修改）
├── 20251234-zs/                # ← 张三的个人目录
├── 20251235-ls/                # ← 李四的个人目录
├── 20251236-ww/                # ← 王五的个人目录
└── ...                         # 每位同学一个独立目录
```

### 个人目录命名规范

> **强制规则**：目录名格式为 `学号-姓名拼音缩写`

| 示例 | 含义 |
|------|------|
| `20251234-zs` | 学号 20251234，张三 |
| `20251235-ls` | 学号 20251235，李四 |
| `20251236-ww` | 学号 20251236，王五 |

- 使用**小写字母**
- 拼音缩写通常取**姓和名首字母**（单名一个字则只取姓首字母 + 名首字母）
- 学号请使用完整学号

### 创建你的个人目录

```bash
# 在仓库根目录下执行（将 20251234-zs 替换为你的实际目录名）
mkdir -p 20251234-zs/{docs,logs}
```

## 个人目录内容要求

每个人在自己的目录下，**必须包含**以下结构：

```
20251234-zs/
├── docs/                       # 【必须】学习笔记与技术文档
│   ├── README.md               #   推荐：本阶段学习内容总览
│   ├── git-learning.md         #   示例：Git 学习笔记
│   └── python-basics.md        #   示例：Python 基础笔记
├── logs/                       # 【必须】周报与进度汇报
│   ├── week01.md               #   第一周周报
│   ├── week02.md               #   第二周周报
│   └── ...
└── code/                       # 【可选】代码练习（可自由组织）
    ├── hello-world/
    └── algorithm-practice/
```

### `docs/` — 学习笔记

- 用于存放你在自学过程中的**学习笔记、技术调研、读书摘要**等
- 建议使用 Markdown 格式（`.md`）
- 推荐添加一个 `docs/README.md` 作为你学习内容的**目录索引**

### `logs/` — 周报

- 每周至少提交一份**学习进度汇报**
- 文件名建议格式：`week01.md`、`week02.md`……
- 周报内容建议包含：
  - 本周学习了哪些内容
  - 遇到了什么问题，如何解决的
  - 下周的学习计划
  - 代码练习完成情况

### `code/` 或其他自定义目录

- 可在个人目录下**自由创建**存放代码的文件夹
- 建议按项目或主题分子目录，保持结构清晰
- 每个代码目录建议附带一个简短的 `README.md` 说明项目用途

## PR 提交流程（标准操作）

每次提交代码或文档，请严格遵循以下流程。养成良好的 Git 习惯，从现在开始。

### 🔄 第 1 步：Sync — 同步上游仓库

在开始新的工作前，**务必**从上游仓库拉取最新代码，避免冲突：

```bash
# 1. 切换到 main 分支（如果你的 fork 默认分支名不同，请对应调整）
git checkout main

# 2. 从上游仓库拉取最新代码
git fetch upstream
git merge upstream/main

# 3. 推送到你自己的远程仓库（可选但推荐）
git push origin main
```

> 💡 每次开始写代码前都做这一步，可以有效避免 PR 冲突。

### ✍️ 第 2 步：Commit — 提交你的改动

```bash
# 查看当前改动了哪些文件
git status

# 只添加你自己的目录（千万不要添加别人的文件！）
git add 20251234-zs/

# 提交 — Message 请简明扼要
git commit -m "docs: 添加 Git 学习笔记"

# 好的 Commit Message 示例：
#   docs: 完成第一周 Python 基础学习笔记
#   feat: 实现冒泡排序和快速排序算法
#   log: 提交第二周学习周报
#   fix: 修复汉诺塔递归的栈溢出问题
```

**Commit Message 规范**（建议遵循 [Conventional Commits](https://www.conventionalcommits.org/)）：

| 前缀 | 用途 | 示例 |
|------|------|------|
| `docs:` | 文档/笔记 | `docs: 添加 Linux 常用命令笔记` |
| `feat:` | 新功能/新代码 | `feat: 完成链表数据结构实现` |
| `fix:` | 修复 Bug | `fix: 修复二分查找边界条件` |
| `log:` | 周报 | `log: 提交第三周学习进度` |
| `refactor:` | 重构代码 | `refactor: 优化快速排序分区逻辑` |

### 🚀 第 3 步：Push & PR — 推送并发起 Pull Request

```bash
# 推送到你自己的远程仓库
git push origin main
```

推送后，打开 GitHub 页面：

1. 进入**你自己的** Fork 仓库页面
2. 点击 **"Contribute"** → **"Open Pull Request"**
3. 确认 base 仓库为 `BobcGn/saike-develop`，base 分支为 `main`
4. 确认 head 仓库为 `你的用户名/saike-develop`，head 分支为 `main`

### 📋 第 4 步：PR 标题与描述规范

**PR 标题**请使用以下格式：

```
[类型] 姓名-具体内容
```

| 示例 | 说明 |
|------|------|
| `[周报] 张三-第一周学习进度` | 提交周报 |
| `[笔记] 张三-Git 入门笔记` | 提交学习笔记 |
| `[代码] 张三-排序算法练习` | 提交代码练习 |
| `[综合] 张三-第一周学习汇总` | 包含多种内容 |

**PR 描述**（可选但推荐）简要说明：
- 本次提交了什么内容
- 有没有特别想让 Reviewer 注意的地方

### ⚠️ 重要提醒

1. **每一份 PR 只能修改你自己的目录。** 我们的 CI 会自动检查——如果你不小心动了别人的文件或仓库根目录配置，PR 会被自动拒绝。
2. **一个 PR 做一件事。** 不要把一周的周报和两周的代码塞进同一个 PR。
3. **先 Sync，再 Push。** 养成这个习惯会让你远离 Git 冲突地狱。

## CI 自动检查说明

当你提交 PR 后，GitHub Actions 会自动运行 **PR Check** 流水线，包含以下检查：

| 检查项 | 说明 | 不通过的后果 |
|--------|------|-------------|
| **目录隔离检查** | 确认你只修改了自己目录下的文件 | ❌ PR 被拒绝，Bot 自动留言说明原因 |
| **Lint 基础检查** | 检查文件格式（行尾空格、文件结尾换行等） | ⚠️ 提示警告，建议修复 |

你可以随时在 PR 页面的 **Checks** 标签页中查看详细的检查日志。

## 常见问题

### Q: 我 Fork 之后，上游仓库更新了怎么办？

执行同步操作即可：

```bash
git checkout main
git fetch upstream
git merge upstream/main
git push origin main
```

### Q: PR 被 CI 拒绝了，说"根目录文件被修改"？

你可能不小心修改了 `.gitignore`、`README.md` 或其他同学的目录。请检查你的改动：

```bash
# 查看你改了哪些文件
git diff upstream/main --name-only
```

如果发现误改，撤销对应文件的改动后重新提交。

### Q: 目录名可以用中文吗？

**不可以。** 目录名必须使用 `学号-拼音缩写` 的格式，全部小写英文和数字。这能避免在不同操作系统上出现编码问题。

### Q: 我的 PR 为什么会和别人的冲突？

通常是因为你没有在提交前 **Sync 上游仓库**。每次开始工作前养成 `git fetch upstream` 的习惯可以避免大部分冲突。

### Q: 我不小心直接 clone 了本仓库并推送了，怎么办？

你会收到 `Permission denied` 错误——因为你对本仓库没有直接写入权限。请按照本文档的第一步，Fork 到自己的账号下再操作。

### Q: 可以提交 `.idea/` 或 `.vscode/` 目录吗？

**不建议。** 这些是 IDE（编辑器）自动生成的本地配置文件，每个同学的开发环境不同。我们已经在 `.gitignore` 中添加了这些文件的忽略规则，CI 也会自动过滤掉它们。建议你在自己的 Fork 仓库中也保持 `.gitignore` 的配置。
